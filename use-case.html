Go-Rent System Use Case Diagram

This use case diagram illustrates the core functionality of the Go-Rent system from the perspective of different stakeholders. It captures how various actors interact with the system's features and displays the relationships between different use cases.

Go-Rent is a peer-to-peer rental platform that allows users to list items for rent (Providers) and rent items from others (Renters). The system facilitates the entire rental process, from listing and searching to booking, payment, and reviews.

UML Notation Legend

  • Actors: Represented as stick figures (Renter, Provider, Admin) or rectangles with «actor» stereotype (Payment System)
  • Use Cases: Represented as ovals inside the system boundary
  • System Boundary: Rectangle containing all use cases
  • Associations: Solid lines connecting actors to use cases they interact with
  • Include Relationships: Dashed arrows with «include» stereotype - one use case includes functionality from another
  • Extend Relationships: Dashed arrows with «extend» stereotype - one use case conditionally extends another
  • Generalization Relationships: Solid lines with triangular arrowheads - one use case is a specialized form of another

Main Actors and Their Roles

Renter

Users who want to rent items from providers. Renters can search for available items, book them for specific time periods, make payments, manage their bookings, write reviews after rental completion, and communicate with providers.

Provider

Users who list their items for rent on the platform. Providers can create and manage item listings, accept or reject booking requests, receive payments for successful rentals, write reviews about renters, and communicate with renters throughout the process.

Admin

System administrators with platform management capabilities. Admins can manage overall system operations, moderate content posted by users (listings, reviews, messages), and generate reports for business analytics.

Payment System

External system that processes payment transactions between renters and providers. This handles the secure transfer of funds, including rental fees, security deposits, and refunds.

Key Use Cases

Common Use Cases

Register and Manage Account - Users can sign up, log in, and manage their personal profiles and account settings.

Renter Use Cases

Search for Rental Items - Browse and filter available items based on various criteria like location, category, price, and availability dates.

Book Item - Request to rent an item for a specific time period.

Make Payment - Complete payment for a booking, including rental fee and security deposit.

Manage Bookings - View, modify, or cancel existing bookings.

Write Review - Provide feedback and ratings for providers and rented items after rental completion.

Communicate with Provider - Send messages to providers regarding bookings, items, or arrangements.

Provider Use Cases

Create Item Listing - Add items to the platform with descriptions, photos, availability calendar, and pricing.

Manage Listings - Edit, update, or remove item listings.

Accept/Reject Booking - Review incoming booking requests and either approve or decline them.

Receive Payment - Collect payments for completed rentals.

Write Review - Provide feedback and ratings for renters after rental completion.

Communicate with Renter - Send messages to renters regarding bookings, items, or arrangements.

Admin Use Cases

Manage System - Oversee platform operations, handle user account issues, and manage system settings.

Moderate Content - Review and manage user-generated content, handle disputes, and ensure platform guidelines are followed.

Generate Reports - Create and analyze reports on platform metrics, user activity, and business performance.

Payment System Use Case

Process Payment - Handle the secure transfer of funds between renters and providers, including payment processing, deposit management, and refunds.

Key Relationships Between Use Cases

Include Relationships

Book Item includes Verify Item Availability - When booking an item, the system must verify the item is available for the requested dates.

Book Item includes Make Payment - The booking process includes making a payment.

Make Payment includes Process Payment - The payment action includes processing through the external payment system.

Process Payment includes Receive Payment - When a payment is processed, the provider receives the payment.

Extend Relationships

Search for Items extends Book Item - After searching, a user may extend the interaction to book an item.

Accept/Reject Booking extends Manage Bookings - When a provider accepts or rejects a booking, this extends the booking management.

Write Review extends Manage Bookings - After a booking is completed, users can extend the interaction to write reviews.

Moderate Content extends various use cases - Admin moderation can extend to content created in listings, reviews, and other user actions.

Understanding Use Case Diagrams

Use case diagrams are a type of behavioral UML diagram that shows the functionality of a system from an external observer's perspective. They help visualize:

  • Who interacts with the system (actors)
  • What functionality the system provides (use cases)
  • How actors and use cases are related
  • Relationships between different use cases

These diagrams are valuable for requirements gathering, communicating with stakeholders, and establishing a clear overview of system functionality without getting into implementation details.
